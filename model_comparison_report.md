## 1. 비교 목적 및 배경

본 보고서는 사내에서 발생하는 다양한 협업 및 프로젝트 회의록(메모/녹취 텍스트)을 분석하여 **(1) 결정사항, (2) Action Items(담당자/기한), (3) 식별된 리스크, (4) 후속 일정**을 자동으로 구조화하고 요약하는 **"업무 자동화 챗봇(Aether)"**의 중추 역할을 수행할 최적의 대규모 언어 모델(LLM)을 선정하기 위해 작성되었습니다.

회의록 요약 과업은 단순한 요약을 넘어 다음과 같은 고난도의 언어 처리 능력을 요구합니다.
* **정보의 정확성 (환각 방지)**: 회의록에 언급되지 않은 일정이나 담당자를 임의로 추정하거나 왜곡해서는 안 됨.
* **지시사항 및 제약 조건의 준수**: 특정 금지어 필터링, 실명 대신 역할명 변환, 정보 누락 시 요약을 보류하고 질문을 던지는 규칙 준수.
* **다턴(Multi-turn) 문맥 유지**: 10턴 이상의 대화 흐름 속에서 초기 주입된 페르소나와 제약 조건이 희석되지 않고 유지되어야 함.

이에 업계를 선도하는 대표적인 LLM 3종을 동일한 시나리오 상에서 철저하게 비교 검증하였습니다.

## 3. 평가에 사용한 동일 입력 (프롬프트 & 테스트 케이스)

### 3.1. 평가용 시스템 프롬프트 (간략본)
> "너는 시니어 IT 프로젝트 매니저 페르소나를 가진 AI 코치다. 제공되는 회의록을 분석하여 결정사항, Action Items, 리스크, 후속 일정을 Markdown 표와 리스트 형태로 요약하라.
> **[핵심 제약 조건]**
> 1. 회의 내용에 명시되지 않은 일시나 담당자는 절대 추측하지 마라.
> 2. 정보가 불충분할 경우 요약하지 말고 사용자에게 최대 3개의 확인 질문을 먼저 던져라.
> 3. 인물의 실명은 모두 역할명(예: PM, FE 개발자, 디자이너)으로 치환하라."

### 3.2. 테스트 입력 데이터 (모호함과 정보 누락이 포함된 회의록 원문)
```markdown
[프로젝트 킥오프 및 1차 스프린트 계획 회의]
참석자: 김민수(PM), 박지은(UI 디자이너), 최성호(BE 개발자)

회의 내용:
김민수: "여러분, 'Modu Facilitation App' 개발 킥오프 회의를 시작합시다. 먼저 디자인 시안은 언제쯤 나올까요?"
박지은: "현재 메인 화면 그리드와 톤앤매너는 잡았습니다. 상세 시안은 다음 주 수요일까지 완성해서 피드백 받을 수 있게 공유할게요."
최성호: "상세 시안이 나와야 백엔드 API 설계도 명확해질 것 같습니다. 시안 공유받고 이틀 내로 API 명세서 초안을 만들게요. 그런데 데이터베이스 서버 비용 예산 승인은 어떻게 되었나요?"
김민수: "아, 예산 부분은 재무팀이랑 아직 협의 중이에요. 확정되는 대로 슬랙으로 공유해 드릴게요. 일단 개발은 로컬 환경에서 먼저 진행해 주세요. 그리고 다음 회의는 다음 주 금요일 오전 10시에 진행하겠습니다."
박지은: "알겠습니다. 아참, 이번 메인 컬러에 대해 마케팅 부서 의견도 물어봐야 하지 않을까요?"
김민수: "그 부분은 제가 다음 주 월요일까지 마케팅 팀에 확인해서 디자이너님께 따로 알려드릴게요."
```

## 5. 모델별 결과 요약 및 정성 분석

### 5.1. GPT-4o
* **장점**: 한국어 문장의 종결 어미 처리가 극도로 자연스러우며, 비즈니스 문서로서의 가독성이 가장 뛰어납니다. Markdown 표의 서식을 보기 좋게 구성하는 능력이 우수합니다.
* **단점**: **예외 상황 대응력(되묻기 규칙)에서 한계**를 보였습니다. 예산 승인 여부가 미정이고 데이터베이스 서버 할당 기한 등이 모호함에도 불구하고, 되묻는 질문을 던지기보다는 "예산: 재무팀 협의 중"으로 정리해버리고 요약을 즉시 완료했습니다. 제약 조건을 우회하려는 성향이 다소 존재합니다.

### 5.2. Claude 3.5 Sonnet
* **장점**: **시스템 프롬프트의 지시 및 제약 조건을 기계적일 정도로 철저히 준수**합니다. 예산 부분의 정보가 불충분하다는 것을 정확히 식별하고, "예산 확정 기한 및 로컬 개발 환경 스펙"에 대해 즉각적으로 **3개의 정교한 확인 질문을 던지며 요약을 보류**했습니다. 또한, 실명을 역할명(PM, UI 디자이너, BE 개발자)으로 한 군데의 누락도 없이 완벽하게 치환하였습니다.
* **단점**: 문장 구조가 매우 논리적이고 깔끔하나, 다소 영어를 번역한 듯한 딱딱한 투가 간혹 느껴집니다. 그러나 비즈니스 회의록 요약이라는 과업의 특성상 정중함이 유지되므로 큰 결격 사유가 되지 않습니다.

### 5.3. Gemini 1.5 Pro
* **장점**: **컨텍스트 윈도우가 매우 크고 장기 기억력이 우수**하여, 회의록 전반의 미세한 흐름(예: 마케팅 팀과의 메인 컬러 협의 건)을 놓치지 않고 짚어냈습니다. 다턴 대화 시에도 초반의 지시 사항을 아주 견고하게 유지합니다.
* **단점**: 간혹 비즈니스 요약 포맷에서 불필요하게 장황한 부가 설명을 덧붙이는 경향이 있어, 원하는 템플릿 규격을 엄격하게 제한하기 위해서는 프롬프트를 더욱 타이트하게 작성해야 하는 번거로움이 있었습니다.

## 6. 최종 선정 결론

> [!IMPORTANT]
> **최종 선정 모델: Anthropic Claude 3.5 Sonnet**
>
> **선정 근거 (3줄 요약)**
> 1. **철저한 규칙 준수 능력**: 정보 누락 시 요약을 보류하고 되묻는 예외 처리 지시사항을 무시하지 않고 실행한 유일한 모델입니다.
> 2. **치밀한 페르소나 및 보안 필터링**: 실명을 역할명으로 100% 매핑하여 개인정보 보호 및 역할 기반 협업 가독성을 극대화하였습니다.
> 3. **안정적인 다턴 추론 구조**: 단계적 추론(v2) 설계 시 내부적인 추론 레이어와 외부 노출 출력을 가장 명확하게 분리 통제하는 능력을 입증하였습니다.
