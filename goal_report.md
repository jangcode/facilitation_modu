# 골 리포트 (Goal Report)

본 문서는 B1-1 미션을 완료한 후, LLM의 작동 메커니즘, 프롬프트 엔지니어링 설계 이론, 모델 선정의 정량적 기준, 그리고 안전한 AI 협업 프로세스 전반에 대해 학습하고 검증한 사항을 종합한 결과 보고서입니다.

---

## 1. LLM의 확률적 생성과 환각(Hallucination)의 원인
- **확률적 생성(Probabilistic Generation) 메커니즘**:
  - LLM은 학습된 방대한 텍스트 데이터로부터 "주어진 컨텍스트 뒤에 올 가장 그럴듯한(확률이 가장 높은) 단어(Token)"를 예측하고 순차적으로 연결하는 방식으로 문장을 구성합니다.
  - LLM은 사실적 진위 여부를 판별하는 외부 데이터베이스가 아니라, 오직 통계적인 연관성과 패턴 매칭에 기반하여 문장을 수학적으로 자아냅니다.
- **환각(Hallucination)이 발생하는 근본 이유**:
  - 확률이 높은 단어들의 연결이 반드시 실제 세계의 참(True)인 사실과 일치하지는 않기 때문입니다.
  - 데이터의 편향, 학습 데이터에 존재하지 않는 지식(Out of Distribution), 또는 정보가 부족함에도 불구하고 그럴듯하게 빈칸을 채워 넣으려는 생성 모델 고유의 특성(Auto-regressive decoding)으로 인해 사실처럼 보이지만 완전히 허구인 정보(환각)가 발행하게 됩니다.

---

## 2. Zero-shot / Few-shot / 단계적 추론 유도(Reasoning Prompt) 기법의 차이와 실무 적용
- **Zero-shot Prompting**:
  - **설명**: 예시 없이 지시어와 대상 입력값만 주고 바로 출력을 요구하는 기법입니다.
  - **용도**: 구조가 단순하거나, LLM이 이미 사전 지식을 바탕으로 아주 높은 확률로 정확하게 답할 수 있는 일반적인 질문(예: 간단한 영한 번역, 대중적인 개념 설명)에 적합합니다.
- **Few-shot Prompting**:
  - **설명**: 프롬프트 내에 입력과 바람직한 출력 쌍(Example)을 최소 1개 이상(대개 3~5개) 포함하여 모델의 출력 스타일, 구조, 제약 조건을 학습시키는 기법입니다.
  - **용도**: 이번 미션의 "모호한 정보 유입 시 질문을 되돌려주는 흐름"처럼, 모델이 단순 지시어로 이해하기 어려운 복잡한 분기 동작이나 일련의 템플릿 서식을 완벽하게 통제하여 출력해야 할 때 유용합니다.
- **단계적 추론 유도 (Reasoning Prompt / Chain of Thought)**:
  - **설명**: 문제를 풀기 전 "단계별로 나누어 먼저 논리적으로 생각하라"고 지시(예: Let's think step by step)하여 논리적 전개력을 끌어올리는 기법입니다.
  - **용도**: 논리 연산, 법률 해석, 복잡한 정책 조건 비교 등 단번에 직관적인 답변을 낼 때 오류가 발생하기 쉬운 복잡한 업무 자동화 과업에 필수적입니다. 최종 결과물에는 불필요하게 장황한 생각 과정이 출력되지 않도록 요약 출력 지침을 병행하여 실무 효율성을 높입니다.

## 3. 평가 축과 객관적 근거 중심의 LLM 모델 비교론
- **"느낌" 기반 평가의 한계**:
  - "대답을 잘하는 것 같다", "말투가 자연스럽다" 등 정성적이고 주관적인 평가는 매 테스트 실행 시점마다 평가 기준이 흔들려 일관된 최적의 인프라를 결정할 수 없게 만듭니다.
- **정량적 평가 축의 설계**:
  - 이번 미션에서는 정확성, 한국어 자연스러움, 긴 문맥(10턴) 유지력, 형식 준수성 등 구체적이고 상호 배타적인 4개 이상의 축을 정의하였습니다.
  - 각 모델에 정확히 동일한 입력 원문과 제약 조건을 투입한 뒤, 각 축에 대해 관찰 가능한 세부 기준(예: 실명 노출 개수, 문맥 유실 턴수 등)을 근거로 삼아 점수(1~5점)를 부여함으로써 최적의 모델(Claude 4.6 Sonnet)을 정량적이고 객관적인 프로세스로 도출할 수 있었습니다.

## 4. 페르소나 및 시스템 프롬프트가 결과 품질에 미치는 영향
- **시스템 프롬프트의 역할**:
  - 사용자 질문(User Prompt)이 입력되기 전, 모델에게 기본적인 배경 규칙과 정체성을 설정하는 최상위 운영 체제 역할을 합니다.
- **페르소나 설정의 실무적 이점**:
  - "IT PM 출신 자동화 코치 정훈"과 같이 페르소나를 명확히 설계하면, 모델의 활성화 파라미터 영역이 해당 도메인의 전문 어휘와 비즈니스 격식 어투 방향으로 조정되어 비즈니스 문서 요약에 필요한 톤앤매너(일관성)를 반영하기 쉬워집니다.
  - 또한 "모르면 모른다고 말하기", "추측성 서술 금지" 등 엄격한 동작 울타리(안전성 및 형식 준수)를 시스템 프롬프트에 내장함으로써 오작동과 환각을 크게 억제합니다.

---

## 5. 생성 결과 검증 및 프롬프트 반복 개선의 프로세스
- **사실 오류 검증 설계**:
  - 생성된 텍스트에서 잘못 인용되거나 임의로 창작된 사실이 있는지 검증하기 위해 회의록 내부 정보를 바탕으로 5개 이상의 질문 세트를 구축하였습니다.
- **Pass/Fail 판정 및 반복 개선 루프**:
  - **Pass 기준**: 모델이 정답을 명확한 근거와 함께 답하거나, 정보가 모호/부재한 경우 임의 추정하지 않고 정확히 "확인 불가"를 명시하는 두 가지 케이스(A/B)를 합격으로 정의했습니다.
  - **Fail 기준**: 근거가 없음에도 아는 것처럼 허위 수치나 기한을 창조하거나, 확인 질문을 생략하는 경우 불합격으로 처리했습니다.
  - 이 검증 결과를 모니터링하여 오류가 발생한 지점(예: 씰 제조사 정보 등)을 확인한 뒤, 시스템 프롬프트 내에 익명화 조건 및 모호성 확인 질문 강제 규칙을 보완하는 반복적인 피드백 루프를 통해 결과물의 재현성과 안정성을 극대화하였습니다.
